# CLAUDE.md

Working conventions for this repo (`PXL-Digital-Application-Samples/pxl-classroom`).

## Git
- **No pull requests.** Commit and push directly to `main`. Never open a PR, never branch "to be safe."
- End commit messages with the standard `Co-Authored-By` trailer.

## Working style
- Be terse and concrete. Do things rather than explain them; give exact values/clicks when the user must act.
- Don't ask for command approval — permissions are set to bypass in `.claude/settings.local.json`.

## Canonical documentation
- **`ARCHITECTURE.md`** — full technical specification (topology, trust model, data model, workflows, actions, flows, constraints). This is the single source of truth for *what the system is*.
- **`RUNBOOK.md`** — operational procedures (setup, onboarding, monitoring, edge cases, recovery).
- **`README.md`** — short orientation + links.
- *Authoritative AI Memory Rule:* when architecture or operational procedure changes, update `ARCHITECTURE.md`, `RUNBOOK.md`, and this file in the same commit. Do not introduce new top-level planning/progress/review documents — the project no longer uses them.

## Project facts (load-bearing — read ARCHITECTURE.md for detail)
- Target platform: **GitHub Team for Education. Never GitHub Enterprise.**
- **Hub-and-spoke:** all workflows live in the central public `pxl-classroom` hub. Per-org control repos hold data only and contain no workflow files.
- **Minimal-minutes (Wave 8):** synchronous provisioning (no queue), one nightly `daily-activity.yml` cron that disables itself when no assignment is active, event-driven dashboard regen. The system bills zero minutes when idle.
- **Admin Panel is the source of truth for lecturer actions** — assignment YAML and override JSON are generated and committed via the SPA + Contents API. The public Page assignments index is stored in the control repo as `assignments.json` keyed by assignment ID (object format), gathered dynamically by the hub build pipeline using `scripts/fetch-pages-data.mjs` to form the global Page data index. `regenerate-dashboard.yml` runs the interim report generator script (`generate-interim-reports.mjs`) to update `reports/dashboard.json` for active assignments in real-time. Roster CSV imports map schema validation (AJV) errors from mechanical formats (e.g. `/students/3/email`) to friendly, line-and-student-specific warnings using `formatRosterValidationError`.
- **`pxl-classroom` CLI is the parallel lecturer surface** (in `cli/`) — same App, same device-flow auth, same schemas as the SPA. Owns the workflows that scale poorly through clicks: CSV roster import, install audits, feedback-PR orchestration, bulk submission download (merging `_manifest.json` conforming to `download-manifest.schema.json` with fields `{login, archive_sha, archive_branch, archive_branch_url, downloaded_at}` by student login instead of overwriting, resetting remote origin to a tokenless URL). The SPA links to it via "Copy CLI command" affordances. For **autograding**, the system supports two paths: lecturer-local (via CLI) or student-side (`github_actions`, synced via SPA).
- **Central GitHub App** is created via the App Manifest form at the SPA's `/setup` route. Frontend deploys automatically via `deploy-frontend.yml` (which aggregates public JSONs from orgs). Setup page includes an interactive verification card that authenticates using the newly created App's Client ID to run audits checking installation and expected/manual permissions (including manual `organization_plan: read` billing scopes) for a specified org.
- **VITE_CORS_PROXY_URL**: If overridden, must explicitly end with `?url=` or `?` (which is auto-rewritten) for the device-flow proxy.
- **Scripting Note:** do not use inline `node -e` scripts in workflow YAML. Extract to `scripts/`.
- **One source of truth per cross-surface concern.** `lib/audit.mjs` exports `EXPECTED_APP_PERMISSIONS` consumed by both CLI and `SetupView.vue`; `lib/gittree.mjs` is HTTP-stack-agnostic and shared by CLI, scripts, and SPA; `lib/control-layout.mjs` exports `CONTROL_SCAFFOLD_DIRS` (the ARCHITECTURE.md §5.1 control-repo layout), consumed by `lib/audit.mjs`'s scaffold check, `scripts/scaffold-control-repo.mjs` (called by `setup-org.yml`), and `control-repo-template/`. Don't fork these. `control-layout.mjs` must stay dependency-free — `setup-org.yml` runs the scaffold script without `npm ci`. `tests/scaffold.test.mjs` fails if any consumer drifts.
- **Dry-run is sacred.** Every CLI `--dry-run` must have zero side effects — no API writes, no PRs, no commits. (`feedback open --dry-run` creating real PRs was a P0; fixed in e967035.)
- **Live Status invariants (AssignmentDetailView):** a post-deadline commit never downgrades a student who has an on-time submission on record (it sets `first_late_sha`, not a `late` status — nightly semantics), a partial refresh is never persisted (all students refreshed, or nothing is committed to the control repo), and grading sync results with 0 check-runs are logged under failed with reason `"no CI run at preserved SHA"`. Sync checks abort on API rate limits/errors. CSV export is generated client-side from the on-screen report, never fetched from the control repo. Live status checks classify based on the current local observation time (`new Date()`) instead of the student-forgeable `committer.date`. Status updates are frozen for students who already have `lock_down_at` set. Grading sync uses a 6-worker pool for concurrency and prevents overwriting pre-existing grades if the graded list is empty. Selective status updates are supported via checkboxes on desktop table/mobile cards, querying only the selected students (falling back to all if none are checked) and clearing selection on completion.
- **Dispatch-and-watch.** SPA workflow triggers are never fire-and-forget: each polls for the artifact/status the workflow produces (publish → polls Pages assignments JSON with a cache buster and 8-minute timeout, daily-activity → `reports/<id>.json`, usage → `usage-latest.json` compared against a `generated_at` baseline, and retry-acceptance → polls the Actions API workflow runs status/conclusion with a 4-minute timeout, prompting the lecturer to confirm if retrying outside the assignment window to trigger with `bypass_window: "true"`). Follow this pattern for any workflow trigger.
- **Lecturer validations & Student gating.** Lecturer actions (granting extensions, retrying acceptances) validate student logins beforehand (checking roster, records, reports, and GitHub user API, returning specific error warnings for typo check vs. not on roster). Extension operations are append-only to preserve overrides history and success toasts clarify that status updates on the next nightly run or live refresh. The student acceptance card is gated client-side on `state === 'published'`, `deadline_at` not passed, and `accepted_count < max_acceptances` (showing closed/cap-reached early/closed status messages instead of Accept button, keeping provisioned repo access if already accepted). Acceptance timeout displays specific causes (cap reached, roster, or GitHub load). The central acceptance handler (`accept.mjs`) verifies that the student is registered in `students/roster.yml` (failing with `rejected:not-on-roster` or `rejected:no-roster` if missing) before recording acceptance or provisioning.
- **SPA UX conventions:** assignment-scoped dates go through `formatDate(iso, assignment.timezone)` (`dashboard.json` entries carry `timezone` for this); editors with unsaved state guard both route-leave and `beforeunload` (incl. the roster tab's pending import — RosterTab exposes `isDirty()`); auth failures render inside the auth card, never the page-level error state, on **every** view; every authenticated view (incl. Admin Panel and assignment detail deep links) shows a sign-in card when no session exists — never a data-shaped empty state; error toasts stay 15 s with hover-pause and a dismiss button (`role="alert"`), displaying text content safely and linking via an optional link property; CSV downloads carry a UTF-8 BOM for Excel; RUNBOOK citations are links to the GitHub blob; per-route document titles live in `router/index.js` `afterEach`; Inter loads non-blocking from `index.html` (no CSS `@import`); router `scrollBehavior` resets page scroll to the top on page navigations.
- **Publish never strands state.** "Save & publish" reverts the assignment YAML to `draft` when the `publish-assignment.yml` dispatch fails, so `state: published` never coexists with a missing broker. Cross-field rules the schema can't express (deadline > opens_at, `max_acceptances` ≥ 1) are enforced in `AdminView.validate()`.
- **New/Edit Assignment Form Invariants (AdminView):** The template repository selector is an autocomplete combobox calling `listOrgTemplates` (GitHub search `is:template` with REST API client-side `is_template` fallback on error/rate-limit) with WAI-ARIA combobox accessibility tags, collapsing on outside click. Editing the repository name pattern sets a manual lock preventing auto-overwrite from title/slug syncs. The default `max_acceptances` cap is `150`. Dynamic field-level validations display red warning messages underneath touched/dirty fields in real-time.
- **CLI destructive ops confirm.** `roster import` prompts before committing removals (TTY) or requires `--force` (non-TTY); `feedback open` separates newly created vs adopted pre-existing PR counts and exits non-zero on partial failure; `--concurrency` parses to positive integers only. Local host runner (`grade --runner host`) prompts for interactive confirmation on TTYs, requiring the `--force-host` bypass flag, or exits non-zero on non-TTY.
- **Git Push Rule:** Always use `$env:GITHUB_TOKEN=""; git push origin main` (PowerShell) when pushing to this repository to avoid authentication failures caused by dummy tokens in the environment.
